# PACS 后端定时 DICOM 同步方案分析

## 1. 背景

当前系统已经具备两类能力：

1. **页面经 HTTP 触发后端操作**
   - 供前端管理页面手工触发远端查询、拉取、推送
   - HTTP 只是入口，不是影像传输协议本身
2. **底层使用标准 DICOM DIMSE**
   - 拉取依赖 `C-FIND + C-MOVE`
   - 推送依赖 `C-STORE`
   - 接收依赖本地 `C-STORE SCP`

因此，如果后续不通过页面，而是希望后端自己定时同步影像，**完全可以继续使用 DICOM 协议**。推荐做法不是“定时调页面接口”，而是由后端定时任务直接调用应用服务，底层继续走 DIMSE。

---

## 2. 结论

**可以使用 DICOM 协议做后端定时拉取/推送，而且这是更合理的实现方式。**

建议保留：

- **HTTP**：只作为页面人工操作入口
- **DICOM DIMSE**：作为真实拉取/推送协议
- **后端定时任务**：直接调用应用服务，不经过 Controller

---

## 3. 可选方案对比

### 方案 A：纯后端定时同步

- 页面不参与同步触发
- 后端任务直接查询远端并执行拉取/推送
- 适合全自动集成场景

### 方案 B：页面手工触发 + 后端定时任务共存

- 页面继续保留人工操作入口
- 后端新增定时任务自动同步
- 两者共用同一套应用服务与 DIMSE 能力

**已选定：方案 B**

原因：

1. 对现有系统改动最小
2. 便于先自动化，再保留人工补偿手段
3. 出问题时可以先人工触发排障，不影响自动任务整体设计

---

## 4. 本次选定方案

### 4.1 方案选择

- **采用方案 B**
- **定时框架采用 `@Scheduled`**

### 4.2 目标形态

保留当前：

- 页面通过 HTTP 调后端进行手工远端操作

新增：

- 后端基于 `@Scheduled` 自动执行影像拉取与推送任务
- 定时任务不经过 Controller，直接调用 Service
- 底层传输仍然使用 DICOM DIMSE

---

## 5. 推荐协议组合

### 5.1 拉取远端影像

**推荐：`C-FIND + C-MOVE`**

流程：

1. 定时任务按配置向远端 PACS 发起 `C-FIND`
2. 根据条件返回 study / series 列表
3. 任务筛选本地未同步或需要补拉的数据
4. 对目标 study 发起 `C-MOVE`
5. 远端将实例通过 `C-STORE` 回传到本地 PACS
6. 本地继续走现有入库流程

### 5.2 推送本地影像到远端

**推荐：本地筛选 + `C-STORE`**

流程：

1. 定时任务筛选本地待推送 study / instance
2. 与远端 AE 建立 DIMSE 连接
3. 逐实例执行 `C-STORE`
4. 记录成功、跳过、失败结果

### 5.3 是否使用 `C-GET`

**当前不建议优先采用 `C-GET``。**

原因：

- 许多 PACS 对 `C-MOVE` 支持更成熟
- `C-FIND + C-MOVE` 更符合当前系统结构
- 可将 `C-GET` 作为后续兼容补充，而不是主路径

---

## 6. 系统设计建议

### 6.1 分层建议

建议新增“同步任务层”，职责如下：

1. **调度层**
   - 基于 Spring `@Scheduled`
   - 决定何时触发拉取/推送

2. **同步应用服务层**
   - 不对外暴露 HTTP
   - 直接调用已有应用服务和 DIMSE 客户端能力

3. **同步状态持久化层**
   - 记录任务执行结果、游标、失败信息、重试信息

### 6.2 推荐调用关系

```text
@Scheduled Job
    -> ScheduledSyncService
        -> RemoteDicomApplicationService.findStudies(...)
        -> RemoteDicomApplicationService.pullFromRemote(...)
        -> DicomApplicationService.findInstances(...)
        -> DicomClientBootstrap.store(...)
```

核心原则：

- 页面和定时任务共用同一套 Service
- Controller 只给页面用
- 定时任务直接调应用服务

---

## 7. 关键设计建议

### 7.1 幂等优先

定时同步必须允许重复执行，不能因为重复数据导致任务失败。

建议保持当前策略：

- 重复 `SOP Instance UID`：识别为已存在并跳过
- 重跑同一批次：不应产生脏数据
- 断点恢复后重试：仍应安全

### 7.2 不要只依赖单点时间戳

不要只用“上次同步时间”作为唯一条件。

建议：

- 每次定时任务扫描一个**回扫窗口**
- 例如扫描最近 2 小时、6 小时、24 小时
- 再依赖本地幂等去重

这样更适合应对：

- PACS 补录
- 晚到数据
- 网络中断后的补同步

### 7.3 按 AE 独立配置策略

每个远端 PACS 可能要求不同，建议按 AE 单独配置：

- 是否启用自动拉取
- 是否启用自动推送
- cron / fixedDelay 配置
- 查询条件模板
- 回扫窗口
- 目标 AET
- 最大并发数
- 重试次数

### 7.4 必须记录同步状态

建议至少记录：

- 任务类型：`PULL` / `PUSH`
- 远端 AET
- 起止时间
- 本次查询条件
- 成功条数 / 跳过条数 / 失败条数
- 最后成功游标
- 最后失败原因

### 7.5 必须防止回流和死循环

这是定时同步里必须优先处理的规则：

- 如果 **PACS B 是从 PACS A 拉取**到的影像，后续 **PACS B 不应再把这批影像自动推送回 PACS A**
- 否则会出现：
  - A -> B 拉取成功
  - B 定时推送又把同一批数据发回 A
  - A/B 之间持续反复同步，形成死循环

建议落地规则：

1. **记录来源 PACS**
   - 每个 study / series / instance 在入库时记录来源
   - 例如：`sourceType=LOCAL_UPLOAD / REMOTE_PULL / REMOTE_PUSH`
   - 以及 `sourceAet=PACS_A`
2. **推送时排除回流目标**
   - 若数据来源是 `REMOTE_PULL`
   - 且 `sourceAet` 与当前推送目标 AET 相同
   - 则本次推送直接跳过
3. **保留人工例外开关**
   - 默认禁止自动回推到来源 PACS
   - 如确有业务需要，再提供人工强制重推选项
4. **同步状态表记录来源链路**
   - 至少记录“这批数据来自哪个 PACS”
   - 后续推送筛选时可直接判断是否需要排除

推荐默认策略：

- **自动拉取的数据，不自动推回其来源 PACS**
- **人工明确触发时，才允许例外**

### 7.6 重试和熔断要明确

建议区分：

- **可重试失败**
  - 网络中断
  - 连接超时
  - 远端临时不可达
- **不可重试失败**
  - 坏文件
  - 非标准对象
  - 配置错误

并增加：

- 连续失败阈值
- 超阈值后自动暂停某 AE 同步
- 通过页面或配置手工恢复

### 7.7 任务不要并发打架

同一个远端 AET 的同方向任务建议避免并发重入。

建议：

- 单机先做本地锁
- 若后续多实例部署，再补分布式锁

## 8. 落地方案菜单

以下菜单用于后续实施时分阶段落地。

### 菜单 1：最小可用版

目标：先把后端自动拉取/推送跑起来。

内容：

1. 增加 `@EnableScheduling`
2. 新增 `ScheduledSyncService`
3. 新增 `PullSyncJob`
4. 新增 `PushSyncJob`
5. 定时任务直接调用现有：
   - `RemoteDicomApplicationService`
   - `DicomApplicationService`
   - `DicomClientBootstrap`
6. 先增加“来源 PACS 排除”规则，避免 A/B 回流死循环
7. 先使用配置文件定义少量固定任务

适合：

- 快速验证自动同步链路
- 先跑通，再完善治理

### 菜单 2：可运营版

目标：具备任务状态、失败记录、可观测性。

在菜单 1 基础上增加：

1. 新增同步任务配置表
2. 新增同步执行记录表
3. 给影像或同步记录增加来源 AET / 来源类型字段
4. 增加任务启停配置
5. 增加最近成功时间、回扫窗口、错误记录
6. 增加页面查看任务执行结果

适合：

- 正式部署到业务环境
- 便于排障与人工补偿

### 菜单 3：增强稳态版

目标：面向更复杂生产环境。

在菜单 2 基础上增加：

1. AE 级别并发控制
2. 失败自动重试
3. 连续失败暂停机制
4. 分布式锁支持
5. 防回流规则白名单 / 黑名单
6. 告警通知
7. 同步速率限制

适合：

- 多 PACS 对接
- 高并发或跨院区同步

---

## 9. 推荐落地顺序

建议按以下顺序推进：

1. **先做菜单 1**
   - 用 `@Scheduled` 跑通后端自动拉取 / 推送
2. **再做菜单 2**
   - 增加状态表和执行记录
3. **最后按需要做菜单 3**
   - 增强生产治理能力

---

## 10. 与当前项目的契合方式

结合当前代码结构，建议新增而不是重写：

### 10.1 保留现状

- `DicomOperationsController`：继续给页面人工触发使用
- `RemoteDicomApplicationService`：继续负责远端查询/拉取
- `DicomClientBootstrap`：继续负责 DIMSE 推送

### 10.2 新增内容

建议新增：

- `ScheduledSyncService`
- `PullSyncScheduler`
- `PushSyncScheduler`
- `SyncPolicyRepository`
- `SyncExecutionRepository`

### 10.3 触发模式

- 页面触发：HTTP -> Controller -> Application Service -> DIMSE
- 定时任务：`@Scheduled` -> ScheduledSyncService -> Application Service -> DIMSE

---

## 11. 最终建议

本项目下一步推荐采用：

- **方案 B**
- **定时任务框架：`@Scheduled`**
- **拉取：`C-FIND + C-MOVE`**
- **推送：`C-STORE`**
- **页面保留人工触发**
- **后端新增自动同步任务**
- **优先保证幂等、回扫窗口、来源追踪、防回流、状态记录、失败重试**

这是当前代码基础上**风险最低、改动最小、最容易演进**的落地方向。
