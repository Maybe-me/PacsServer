# 阅片 V2 设计评审要点

## 1. 评审目标

本文件用于在 Viewer V2 开发前和开发过程中，统一评审重点，避免只做“换皮”，而忽略真正影响阅片体验的问题。

## 2. 核心评审维度

### 2.1 工作区信息架构

关注点：

- 顶部、左侧、中部、右侧分区是否职责清晰
- 是否能支持连续阅片而不是频繁跳转
- 是否保留必要诊断信息但不过度干扰主流程

风险：

- 信息太多，继续保持“调试台”感觉
- 信息太少，丢失当前已具备的兼容诊断价值

### 2.2 视口工作流

关注点：

- 活动视口是否足够醒目
- 多视口切换是否直观
- viewport 顶部状态、覆盖层、降级提示是否清楚

风险：

- 多视口状态不明确，用户容易误操作
- 降级信息隐藏过深，黑屏排查效率下降

### 2.3 工具栏设计

关注点：

- 工具分组是否符合阅片习惯
- 主操作与辅助操作是否分层
- 布局切换、cine、滚动、重置等入口是否合理

风险：

- 工具条仍像开发调试工具集合
- 高价值操作入口不明显

### 2.4 导航与缩略图

关注点：

- Series 浏览是否比当前表格更适合连续查看
- Thumbnail 是否能表达实例状态、坏帧、不可兼容等信息
- 浏览层级是否清晰

风险：

- 只做视觉调整，不改浏览模型
- 缩略图区域继续承担过多文本诊断内容，影响扫读

### 2.5 兼容性与稳定性

关注点：

- V2 改版不能破坏当前黑屏修复成果
- 现有坏帧跳过、错误捕获、兼容性提示必须保留
- WADO、多帧、失败帧、浮点像素数据等兼容保护要继续可见

风险：

- UI 改版过程中把现有稳定性逻辑绕坏
- 为了“像工作站”而把真实错误状态隐藏掉

## 3. 视觉风格约束

Viewer V2 可以参考 OHIF 的设计思路，但必须遵守以下约束：

- 不逐像素复制 OHIF 的页面结构和样式
- 不直接照搬其图标编排、配色细节或布局比例
- 允许参考其“工作站式”信息组织方式
- 最终产物必须保持项目自有风格

## 4. 开发前检查项

在进入 Viewer V2 编码前，建议先确认：

1. 当前黑屏问题是否已足够稳定
2. 现有 store / viewport / toolbar 能力是否能复用
3. 哪些诊断信息应下沉到右侧上下文面板
4. 哪些现有表格型模块需要直接替换而不是修补

## 5. 交付评审标准

Viewer V2 第一版在评审时，至少应满足：

- 有明显工作站式阅片体验提升
- 保留当前 viewer 的稳定性与兼容诊断能力
- UI 结构清晰，不再是 debug-first 页面
- 风格参考 OHIF，但不是复制品
