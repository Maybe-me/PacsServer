# 代码评审结论

本文档用于汇总当前仓库状态下较值得关注的风险与注意事项。

## 一、高优先级问题

### 安全相关

1. **默认 JWT Secret**
   - Java 配置里仍存在默认值 `change-me`
   - 正式环境必须覆盖

2. **CORS 过宽**
   - 当前审阅到的配置偏开放
   - 生产环境应限制来源与允许的方法

3. **管理接口权限保护不明显**
   - 当前看到的 Controller 中没有明显的方法级权限注解
   - 如果上层安全未接入或被关闭，风险较高

4. **DICOM TLS 默认关闭**
   - DICOM 端口默认是明文传输
   - 对正式环境不够安全

### Viewer / 前端

5. **前端缺少自动化测试**
   - 当前主要依赖构建通过和人工验证

6. **Cornerstone worker 设置偏保守**
   - 对大体量多帧序列，性能可能不是最优

7. **Viewer 黑屏收尾还未完全结束**
   - 主流程已通
   - 但剩余异常样本还需要继续收口

8. **大数据量列表的 UX 风险**
   - 还没有明显的分页或虚拟滚动策略

### Java 后端 / 运维

9. **调度依赖数据库锁**
   - 数据库或锁链路异常时，同步任务会受影响

10. **存储抽象仍以本地文件为主**
   - 抽象已经有了
   - 但当前明显完成度最高的是本地存储路径

11. **高可用与优雅停机能力未形成完整说明**
   - 当前更像单实例 DICOM 服务形态

### Python Generator

12. **任务状态保存在内存中**
   - 长时间运行时需要考虑清理策略

13. **存在较宽泛的异常处理**
   - 后续可继续细化错误分类与诊断

14. **Tag override 的类型校验较弱**
   - 灵活性高，但强校验不足

15. **存在未明显使用的依赖**
   - `httpx` 在当前审阅中没有看到明确运行时使用

## 二、当前做得比较好的部分

1. **后端分层相对清晰**
   - domain / application / infrastructure / boot 边界明确

2. **Java 集成测试覆盖较完整**
   - 已覆盖 ingest、QIDO/WADO、sync、Python generator 联动

3. **Viewer 稳定性提升明显**
   - 已补上多帧处理、兼容性分类、渲染诊断、运行时恢复

4. **部署基础已经具备**
   - 已有 Docker、compose、Flyway、profile 配置

## 三、建议的后续动作

1. 继续补强管理接口和 DICOMweb 的正式环境安全方案
2. 给前端补一层最基本的自动化验证
3. 继续推进唯一剩余项 `viewer-black-screen-followup`
4. 继续梳理同步任务、存储增长、generator 任务留存的运维策略
